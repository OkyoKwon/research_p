# 크립토 결제 서비스 시나리오 분석 보고서

4개 크립토 결제 서비스(Base Pay, Binance Pay, Coinbase Pay, Stripe Crypto)의 결제/정산/환불 시나리오를 구조화한 종합 분석 문서이다.

- **작성 일시**: 2026-04-15
- **분석 대상**: Base Pay, Binance Pay, Coinbase Commerce, Stripe Crypto
- **분석 관점**: 결제(Payment) - 정산(Settlement) - 환불(Refund) 시나리오

---

# 1. Base Pay (Commerce Payments Protocol)

Coinbase가 자체 L2 블록체인 Base 네트워크 위에 구축한 온체인 결제 프로토콜이다. 전통 결제의 Authorize-Capture 모델을 스마트 컨트랙트로 구현한 점이 핵심 차별점이다.

---

## 1.1 결제(Payment) 시나리오

### 1.1.1 즉시 결제 (Charge / Direct Transfer)

- **시나리오 이름**: 즉시 결제 (Charge)
- **참여자**: 구매자, 머천트, 오퍼레이터(Coinbase), 스마트 컨트랙트(Transfers.sol / AuthCaptureEscrow)
- **단계별 흐름**:
  1. 머천트가 결제 요청 생성 (API 또는 Shopify 플러그인)
  2. 오퍼레이터가 `TransferIntent`(서명된 결제 원시자료) 생성 -- 금액, 머천트 주소, 만료 기한, 수수료 포함
  3. 구매자가 지갑 연결 후 결제 승인
  4. 스마트 컨트랙트 실행: 구매자 토큰과 머천트 정산 토큰이 다르면 Uniswap V3를 통해 자동 스왑, Permit2 또는 EIP-3009 서명으로 토큰 전송
  5. 머천트 지갑에 정산 토큰 직접 전송 + 오퍼레이터 수수료 동시 차감
  6. `Transferred` 이벤트 발행 (온체인 기록)
- **수수료/시간 정보**:
  - 플랫폼 수수료: 1% (정산 금액에서 차감, 머천트 부담)
  - 네트워크 가스비: ~$0.01 (Base 기준, 구매자 부담)
  - DEX 스왑 슬리피지: 변동 (토큰이 다를 경우 구매자 부담)
  - 전체 소요 시간: Base 기준 약 2초
- **특이사항/제약사항**:
  - 에스크로 없이 즉시 머천트에게 전송됨
  - Uniswap V3에 유동성이 있는 모든 토큰으로 결제 가능
  - 디지털 상품, 구독 활성화, 즉시 배송 상품에 적합
  - 1세대(Transfers.sol) 9가지 결제 함수 및 2세대(AuthCaptureEscrow) `charge()` 함수 모두 지원

### 1.1.2 인가-캡처 결제 (Authorize-Capture)

- **시나리오 이름**: 에스크로 기반 인가-캡처 결제
- **참여자**: 구매자, 머천트, 오퍼레이터(Coinbase/Shopify), AuthCaptureEscrow 스마트 컨트랙트, Token Store(에스크로)
- **단계별 흐름**:
  1. 머천트/오퍼레이터가 `PaymentInfo` 생성 -- `maxAmount`, `authorizationExpiry`, `refundExpiry`, `minFeeBps`~`maxFeeBps` 등 포함
  2. 구매자가 결제 승인 (지갑 서명)
  3. **Authorize**: Token Collector가 구매자 자금을 에스크로(Token Store)에 예치. Token Store는 오퍼레이터별 격리된 보관소(CREATE2 배포). `authorizationExpiry` 타이머 시작
  4. **[대기 기간]**: 머천트가 주문 확인, 재고 확인, 배송 준비 등. 이 기간 동안 Void(취소) 가능
  5. **Capture**: 오퍼레이터가 에스크로에서 머천트로 자금 전송. 부분 캡처(partial capture) 지원. 총 캡처 금액 <= 승인 금액(maxAmount). `authorizationExpiry` 이전에만 실행 가능
  6. 미캡처 잔여분은 구매자에게 자동 반환되거나 `reclaim()` 가능
- **수수료/시간 정보**:
  - 플랫폼 수수료: minFeeBps~maxFeeBps (일반적으로 1%), Capture 시 차감
  - 네트워크 가스비 (Authorize): ~$0.01
  - 네트워크 가스비 (Capture): ~$0.01
  - Void 시 수수료: 없음
- **특이사항/제약사항**:
  - 업계 유일의 온체인 에스크로 기반 Authorize-Capture 프로토콜
  - 부분 캡처 지원 (다중 배송, 분할 배송 시 유용)
  - Token Store는 Minimal Proxy 패턴으로 가스 효율 최적화
  - 비업그레이드 컨트랙트로 배포 후 변경 불가 (보안 보장)
  - 실물 상품 이커머스, 예약/호텔 결제, 세금 확정 후 결제에 적합

### 1.1.3 x402 머신 결제

- **시나리오 이름**: HTTP 402 기반 머신/AI 에이전트 결제
- **참여자**: 클라이언트(AI 에이전트 또는 사용자), 서버(API 제공자), Facilitator(Stripe 등), 블록체인(Base/Solana)
- **단계별 흐름**:
  1. 클라이언트가 유료 리소스에 HTTP GET 요청
  2. 서버가 **HTTP 402 Payment Required** 응답 + 결제 요건(Invoice) 반환 -- 금액, 수신 주소, 유효 기간, 결제 스킴(`exact` 또는 `upto`) 포함
  3. 클라이언트가 USDC로 결제 서명 생성 (EIP-3009, Permit2, 또는 ERC-7710)
  4. 클라이언트가 `PAYMENT-SIGNATURE` HTTP 헤더에 서명을 포함하여 재요청
  5. 서버(또는 Facilitator)가 결제 검증: 서명 복원, 잔고 확인, 금액 일치, 유효 기간 확인, `transferWithAuthorization()` 시뮬레이션 실행
  6. 검증 성공 시 온체인 정산 + 리소스 제공 (HTTP 200)
- **수수료/시간 정보**:
  - Stripe x402 Facilitator: 수수료 0% (월 1,000건 무료, 이후 미공개 과금)
  - 네트워크 가스비: ~$0.01 (Base)
  - 전체 소요 시간: ~2초 (Base)
- **특이사항/제약사항**:
  - 두 가지 결제 스킴: `exact`(고정 금액) / `upto`(사용량 기반 최대 금액)
  - USDC 점유율 98.7%
  - AI 에이전트 API 과금, 유료 API pay-per-request, IoT 마이크로페이먼트에 적합
  - x402 Foundation 거버넌스: Google, AWS, Visa, Circle, Anthropic 참여
  - 연간화 거래량 ~$600M (2026.03 기준)

## 1.2 정산(Settlement) 시나리오

### 1.2.1 온체인 즉시 정산 (Charge / Direct Transfer)

- **시나리오 이름**: 즉시 결제 온체인 정산
- **참여자**: 구매자, 머천트, 스마트 컨트랙트, Uniswap V3 (자동 스왑 시)
- **단계별 흐름**:
  1. 구매자 결제 실행
  2. 스마트 컨트랙트가 원자적으로 머천트 지갑에 정산 토큰 전송
  3. 동시에 오퍼레이터 수수료 차감
- **수수료/시간 정보**:
  - Base 기준 약 2초 내 정산 완료
  - 정산 수수료: 1% (결제 수수료에 포함)
- **특이사항/제약사항**:
  - 정산 통화: USDC(기본), ETH, 기타 토큰 (머천트 선호에 따라 설정)
  - 자동 변환: Uniswap V3 DEX를 통해 결제 토큰 -> 정산 토큰 자동 변환

### 1.2.2 에스크로 정산 (Authorize-Capture)

- **시나리오 이름**: 에스크로 캡처 후 정산
- **참여자**: 오퍼레이터, 머천트, Token Store(에스크로), AuthCaptureEscrow 컨트랙트
- **단계별 흐름**:
  1. Authorize 시 자금이 Token Store(에스크로)에 보관
  2. 머천트 확인 후 Capture 실행 시 에스크로에서 머천트로 전송
  3. 부분 캡처 가능 (다중 배송 등)
- **수수료/시간 정보**:
  - Capture 실행 시 즉시 정산
  - authorizationExpiry 이전에만 Capture 가능
- **특이사항/제약사항**:
  - 미캡처 잔여분은 구매자가 `reclaim()` 호출로 회수 가능

### 1.2.3 법정화폐 정산 (간접 경로)

- **시나리오 이름**: 법정화폐 전환 정산
- **참여자**: 머천트, Coinbase Business/Exchange, 은행
- **단계별 흐름**:
  1. **Coinbase Business 경유** (미국/싱가포르 법인만): USDC 수령 -> Coinbase Business 내 fiat off-ramp -> 은행 출금
  2. **Coinbase 거래소 경유** (글로벌): 머천트 지갑의 USDC를 Coinbase 거래소로 전송 -> USDC -> USD 전환 후 은행 출금
  3. **Shopify 통합 경유**: Shopify가 Stripe 인프라를 활용하여 현지 통화로 정산
- **수수료/시간 정보**:
  - Coinbase Business/Exchange 경유: 추가 1-5 영업일 소요
  - Shopify 경유 정산 수수료: 0% (Shopify 측)
- **특이사항/제약사항**:
  - 법정화폐 직접 은행 정산 미지원 (경쟁사 대비 구조적 약점)
  - BitPay(38개국), Stripe Stablecoin(USD 직접 정산) 대비 열위

## 1.3 환불(Refund) 시나리오

### 1.3.1 캡처 전 취소 (Void)

- **시나리오 이름**: 에스크로 전액 취소 (Void)
- **참여자**: 오퍼레이터, 구매자, AuthCaptureEscrow 컨트랙트, Token Store
- **단계별 흐름**:
  1. 머천트/오퍼레이터가 환불 사유 확인 (재고 부족, 주문 취소 등)
  2. 오퍼레이터가 `void()` 함수 호출
  3. AuthCaptureEscrow가 Token Store에 보관된 전체 잔여 승인 금액을 구매자에게 즉시 반환
  4. 결제 상태: Authorized -> Voided
- **수수료/시간 정보**:
  - 플랫폼 수수료: 발생하지 않음 (Capture가 실행되지 않았으므로)
  - 네트워크 가스비: ~$0.01 (Void 트랜잭션)
  - Base 기준 약 2초 내 완료
- **특이사항/제약사항**:
  - 결제당 1회만 실행 가능
  - 남은 전체 승인 금액이 반환됨 (부분 Void 불가)
  - Capture가 이미 실행된 부분에 대해서는 Void 불가

### 1.3.2 캡처 후 환불 (Refund)

- **시나리오 이름**: 캡처 후 환불 (전액/부분)
- **참여자**: 오퍼레이터(또는 머천트), 구매자, OperatorRefundCollector, AuthCaptureEscrow 컨트랙트
- **단계별 흐름**:
  1. 환불 사유 발생 (반품, 불만족 등)
  2. 오퍼레이터(또는 머천트)가 `refund()` 함수 호출
  3. OperatorRefundCollector가 다양한 소스에서 환불 자금 조달 (머천트 잔고, 오퍼레이터 잔고, 별도 지정 주소)
  4. 환불 자금이 구매자에게 전송
- **수수료/시간 정보**:
  - 네트워크 가스비: ~$0.01 (환불 트랜잭션)
  - 플랫폼 수수료(1%)의 환불 여부: 오퍼레이터 재량 (일반적으로 미환불)
  - Base 기준 약 2초 내 완료
- **특이사항/제약사항**:
  - 부분 환불 지원 (2세대 프로토콜)
  - 환불 토큰은 원래 결제 토큰과 동일
  - `refundExpiry` 이후에는 환불 불가 (최종 확정 상태)
  - 가맹점이 환불 가스비를 위한 별도 토큰 잔고를 보유해야 함

### 1.3.3 인가 만료 후 소비자 회수 (Reclaim)

- **시나리오 이름**: 에스크로 자금 소비자 직접 회수
- **참여자**: 구매자, AuthCaptureEscrow 컨트랙트, Token Store
- **단계별 흐름**:
  1. Authorize 완료 후 `authorizationExpiry`가 경과
  2. 머천트/오퍼레이터가 Capture를 실행하지 않음
  3. 구매자가 직접 `reclaim()` 함수 호출
  4. AuthCaptureEscrow가 Token Store의 전체 에스크로 금액을 구매자에게 반환
  5. 결제 상태: Authorized -> Reclaimed
- **수수료/시간 정보**:
  - 네트워크 가스비: ~$0.01
- **특이사항/제약사항**:
  - 중앙화된 중재 기관 없이도 구매자가 자금을 회수할 수 있는 온체인 안전장치
  - `authorizationExpiry`는 PaymentInfo에 사전 정의
  - 오퍼레이터/머천트 오작동 시 구매자의 최종 보호 수단

### 1.3.4 즉시 결제(Charge) 후 환불

- **시나리오 이름**: Charge 완료 후 환불
- **참여자**: 머천트, 구매자, OperatorRefundCollector (2세대)
- **단계별 흐름**:
  1. Charge(즉시 결제)가 완료되어 머천트에게 자금이 이미 전송됨
  2. 환불 사유 발생
  3. **1세대(Transfers.sol)**: 프로토콜 외부에서 처리 -- 머천트가 별도 온체인 트랜잭션으로 구매자에게 직접 전송
  4. **2세대(AuthCaptureEscrow의 Charge)**: `refund()` 함수를 통해 환불 가능 (refundExpiry 이내). OperatorRefundCollector를 통해 자금 조달. 부분 환불 지원
- **수수료/시간 정보**:
  - 가스비 별도 발생 (~$0.01 on Base)
  - 1세대: 머천트가 직접 전송하므로 가스비 머천트 부담
- **특이사항/제약사항**:
  - 1세대 프로토콜은 환불 자동화 미지원
  - 2세대 프로토콜에서 refundExpiry 이후 환불 불가

### 1.3.5 분쟁 해결

- **시나리오 이름**: 온체인 결제 분쟁 해결
- **참여자**: 구매자, 머천트, 오퍼레이터(Coinbase)
- **분쟁 시나리오별 해결 경로**:
  - 캡처 전 분쟁: Void 또는 authorizationExpiry 경과 후 Reclaim
  - 캡처 후 머천트 환불 동의: Refund (OperatorRefundCollector)
  - 캡처 후 머천트 환불 거부: 오퍼레이터(Coinbase) 정책적 중재 -- 프로토콜 외부 처리
  - 사기 거래: 구매자가 강제할 수 있는 중앙화된 중재 기관 부재
- **특이사항/제약사항**:
  - 신용카드의 차지백(chargeback) 메커니즘 없음
  - 머천트에게 유리 (무단 차지백 리스크 제거)
  - 구매자 보호: 에스크로(Auth-Capture)가 부분적으로 보완하나 캡처 이후 제한적

# 2. Binance Pay

Binance 거래소 기반 오프체인 암호화폐 결제 서비스이다. Binance 내부 원장(Binance Ledger)에서 모든 자금 이동을 처리하여 가스비 제로, 즉시 정산을 달성한다.

---

## 2.1 결제(Payment) 시나리오

### 2.1.1 P2P 개인 간 결제

- **시나리오 이름**: Binance 사용자 간 P2P 송금
- **참여자**: 송금자(Binance 사용자), 수신자(Binance 사용자), Binance 내부 원장
- **단계별 흐름**:
  1. 송금자가 Binance 앱 > Pay 메뉴 진입
  2. 상대방의 Pay ID, 이메일, 전화번호 입력 또는 QR 코드 스캔
  3. 전송할 암호화폐 종류 선택 (300종 이상 지원)
  4. 전송 금액 입력
  5. 확인 후 즉시 상대방 Binance Funding Wallet에 이체 완료
- **수수료/시간 정보**:
  - 수수료: 0% (무료)
  - 처리 속도: 즉시 (오프체인)
  - 블록체인 가스비: 0% (블록체인 트랜잭션이 아님)
- **특이사항/제약사항**:
  - 송수신 양측 모두 Binance 계정 + KYC 인증 필수
  - 300종 이상 암호화폐 지원
  - 활성 P2P 사용자: 4,500만 명+

### 2.1.2 Merchant 결제 (Hosted Checkout)

- **시나리오 이름**: Binance 호스팅 결제 페이지를 통한 가맹점 결제
- **참여자**: 가맹점 서버, 고객(Binance 사용자), Binance Pay API, Binance 내부 원장, FX Engine
- **단계별 흐름**:
  1. 가맹점 서버가 Create Order V2 API (`POST /binancepay/openapi/v2/order`) 호출하여 주문 생성
  2. API 응답으로 `checkoutUrl`, `qrcodeLink`, `deeplink`, `universalUrl` 수신
  3. 고객을 Binance Web Checkout Page로 리다이렉트
  4. 고객이 3가지 방법 중 선택: QR 코드 스캔 / Binance 앱 열기(딥링크) / 웹 로그인
  5. 고객이 50종 이상의 암호화폐 중 결제 수단 선택
  6. 고객이 결제 확인
  7. Binance Pay 시스템이 결제 처리 + 자동 환전(필요 시)
  8. 가맹점 Binance Funding Wallet에 정산 통화(USDT 등)로 즉시 입금
  9. Binance Pay가 가맹점에 Webhook 콜백 발송 (`bizStatus: "PAY_SUCCESS"`)
  10. 고객이 `returnUrl`로 리다이렉트
- **수수료/시간 정보**:
  - 가맹점 결제 수수료 (MDR): 1.0%
  - FX 스프레드 (숨겨진 비용): 추정 0.1~0.5%
  - 블록체인 가스비: 0% (오프체인)
  - 처리 속도: 즉시
- **특이사항/제약사항**:
  - PCI DSS 부담 없음
  - 고객은 Binance 계정 필수
  - 가장 빠른 통합 방법

### 2.1.3 Merchant 결제 (C2B Native API)

- **시나리오 이름**: 가맹점 자체 UI를 통한 결제
- **참여자**: 가맹점 서버, 고객(Binance 사용자), Binance Pay API
- **단계별 흐름**:
  1. 가맹점이 Create Order V2 API 호출
  2. 응답의 `qrContent`를 활용하여 가맹점 자체 UI에서 QR 코드 렌더링
  3. 또는 `deeplink`를 활용하여 인앱 결제 흐름 구현
  4. 고객이 Binance 앱에서 결제 수단 선택 후 결제
  5. Webhook으로 결제 결과 수신
- **수수료/시간 정보**:
  - MDR: 1.0%
  - 가스비: 0%
- **특이사항/제약사항**:
  - 대규모 이커머스/앱 서비스에 적합
  - 개발 난이도 높음

### 2.1.4 Payment Links 결제

- **시나리오 이름**: 코딩 없는 결제 링크 생성
- **참여자**: 가맹점, 고객(Binance 사용자), Binance Merchant Management Platform
- **단계별 흐름**:
  1. 가맹점이 Merchant Management Platform에서 결제 링크 생성 (코딩 불필요)
  2. 링크를 이메일, SNS, 웹사이트 등에 공유
  3. 고객이 링크 클릭 후 Binance 결제 플로우 진행
  4. 결제 완료 시 가맹점 대시보드에서 확인
- **수수료/시간 정보**:
  - MDR: 1.0%
- **특이사항/제약사항**:
  - 소규모 사업자/프리랜서에 적합
  - 개발 난이도 제로

### 2.1.5 QR 코드 오프라인 결제

- **시나리오 이름**: 오프라인 매장 QR 결제
- **참여자**: 가맹점(매장), 고객(Binance 사용자)
- **단계별 흐름**:
  1. 가맹점이 API로 주문 생성 후 QR 코드를 매장 디스플레이에 표시
  2. 고객이 Binance 앱의 스캐너로 QR 코드 스캔
  3. 앱이 자동으로 금액, 결제 대상 정보를 인식
  4. 고객이 결제할 암호화폐 선택 후 확인
  5. 즉시 처리 완료, 가맹점에 Webhook 알림
- **수수료/시간 정보**:
  - MDR: 1.0%
  - 처리 속도: 즉시
- **특이사항/제약사항**:
  - 오프라인 리테일에 적합

### 2.1.6 Batch Payout (일괄 출금)

- **시나리오 이름**: 가맹점이 다수에게 일괄 출금
- **참여자**: 가맹점, 수신자(최대 500명/배치), Binance Pay Payout API
- **단계별 흐름**:
  1. 가맹점이 Payout API (`POST /binancepay/openapi/payout/transfer`) 호출 또는 CSV 업로드
  2. Binance 사용자 수신: Funding Wallet에 즉시 입금
  3. 비 Binance 사용자 수신: 이메일 초대 발송
  4. 비 Binance 사용자: 72시간 내 KYC 미완료 시 자동 환불
- **수수료/시간 정보**:
  - Payout 수수료: 0.80% (건당 최대 USD 5)
- **특이사항/제약사항**:
  - 배치당 최대 500명
  - 2024년 12월부터 무료에서 유료 전환

## 2.2 정산(Settlement) 시나리오

### 2.2.1 오프체인 즉시 정산

- **시나리오 이름**: Binance 내부 원장 즉시 정산
- **참여자**: 고객, Binance Internal Ledger, FX Engine, 가맹점
- **단계별 흐름**:
  1. 고객 잔고에서 결제 암호화폐 차감 (원장 기록)
  2. FX Engine: 결제 암호화폐 -> 정산 통화 변환 (내부 유동성 풀 활용, 스프레드 내재)
  3. MDR 1% 차감 (Binance 수수료 계정)
  4. 가맹점 잔고에 정산 통화 가산 (원장 기록)
  5. 가맹점 Binance Funding Wallet에 즉시 입금
- **수수료/시간 정보**:
  - 정산 속도: 즉시 (T+0 실시간)
  - MDR: 1%
  - FX 스프레드: 추정 0.1~0.5%
  - Binance Ledger 성능: 10,000+ TPS, 10ms 이내 지연
- **특이사항/제약사항**:
  - 기본 정산 통화: USDT
  - 가맹점이 정산 설정에서 선호 통화를 정의 가능
  - 블록체인을 거치지 않는 오프체인 처리

### 2.2.2 법정화폐 전환 경로 (간접)

- **시나리오 이름**: 법정화폐 전환 및 은행 출금
- **참여자**: 가맹점, Binance 거래소, 채널 파트너(Alchemy Pay 등)
- **단계별 흐름**:
  1. 결제 수취 -> USDT로 자동 정산
  2. Binance 거래소에서 수동 환전 (거래소 수수료 0.1%)
  3. 법정화폐 출금 (출금 수수료 별도)
- **수수료/시간 정보**:
  - 거래소 환전 수수료: 0.1%
  - 법정화폐 출금 수수료: ~$5~15
  - 총 실효 수수료율: ~1.4~1.5%
- **특이사항/제약사항**:
  - 법정화폐 직접 은행 입금 불가 (최대 구조적 약점)
  - 수동 처리 필요
  - 우회 경로: Alchemy Pay 등 채널 파트너 경유 가능 (추가 수수료)

## 2.3 환불(Refund) 시나리오

### 2.3.1 전액 환불

- **시나리오 이름**: Merchant 결제 전액 환불
- **참여자**: 가맹점, 고객(Binance 사용자), Binance Pay 시스템
- **단계별 흐름**:
  1. 가맹점이 Merchant Management Platform > Transaction Tab에서 해당 주문 확인
  2. 또는 Refund Order API (`POST /binancepay/openapi/order/refund`) 호출
  3. 원래 주문의 `prepayId`와 환불 금액(= 원래 주문 금액) 입력
  4. 환불 사유 입력 (선택)
  5. Binance Pay가 원래 주문 검증 + 환불 금액 검증 + 중복 요청 확인
  6. 가맹점 Binance Funding Wallet에서 환불금 차감
  7. 고객 Binance Funding Wallet에 환불금 즉시 입금
  8. MDR 수수료도 비례 환불 (`refundCommission`)
  9. Webhook으로 `bizStatus: "REFUND_SUCCESS"` 발송
  10. 주문 상태가 `FULL_REFUNDED`로 변경
- **수수료/시간 정보**:
  - 환불 처리 속도: 즉시 (오프체인)
  - 수수료 환불 공식: `refundCommission = (refundAmount / orderAmount) * originalCommission`
- **특이사항/제약사항**:
  - API + 대시보드 양방향 환불 지원
  - MDR 수수료 비례 환불됨

### 2.3.2 부분 환불

- **시나리오 이름**: Merchant 결제 부분 환불
- **참여자**: 가맹점, 고객(Binance 사용자), Binance Pay 시스템
- **단계별 흐름**:
  1. 가맹점이 Refund Order API 호출 (refundAmount < orderAmount)
  2. Binance Pay가 검증 후 부분 환불 처리
  3. 여러 차례 반복 가능 (누적 환불 금액 <= 원래 주문 금액)
- **수수료/시간 정보**:
  - 환불 처리 속도: 즉시
- **특이사항/제약사항**:
  - `remainingAttempts`가 1이면 다음 환불 시 자동으로 전액 환불 처리
  - `duplicateRequest` 플래그로 중복 요청 감지
  - 주요 에러 코드: `400607`(환불 횟수 초과), `400608`(잘못된 환불 금액), `400611`(가맹점 잔액 부족)

### 2.3.3 환불 통화 처리

- **시나리오 이름**: 환불 통화 변환 규칙
- **참여자**: 가맹점, 고객
- **핵심 규칙**:
  - 환불은 가맹점이 수취한 정산 통화로 처리 (역환전 미수행)
  - 고객이 BTC로 결제, 가맹점이 USDT로 정산 받은 경우 -> 고객에게 USDT로 환불
- **특이사항/제약사항**:
  - 사용자 불만: BTC로 결제했는데 USDT로 환불 받으면, BTC 가격 상승 시 고객 손실 발생

### 2.3.4 분쟁 해결

- **시나리오 이름**: 결제 분쟁 해결 절차
- **참여자**: 가맹점, 고객, Binance 고객지원팀
- **Merchant 결제 분쟁**:
  - 차지백(chargeback) 메커니즘 없음
  - 환불 여부는 가맹점 재량
  - Binance는 중개자로서 분쟁 조정 가능하나 강제력 제한
  - 구매자 보호 프로그램 없음
- **P2P 거래 분쟁**:
  1. 항소(Appeal) 개시
  2. 상대방에게 10분간 채팅 협의 기회 부여
  3. 합의 미달 시 Binance 고객지원팀 중재
  4. 24~48시간 내 응답
- **수수료/시간 정보**:
  - P2P 분쟁 처리 수수료: 4번째 분쟁부터 부과 (처음 3회 무료)

# 3. Coinbase Commerce (Coinbase Pay)

가맹점이 암호화폐로 결제를 수령할 수 있는 결제 게이트웨이 서비스이다. 비수탁형(non-custodial) 아키텍처가 특징이며, Commerce Payments Protocol로 전환 중이다.

---

## 3.1 결제(Payment) 시나리오

### 3.1.1 기본 결제 (Coinbase Commerce Charge)

- **시나리오 이름**: Commerce API 기반 즉시 결제
- **참여자**: 가맹점, 소비자, Commerce API, DEX(Uniswap V3)
- **단계별 흐름**:
  1. 가맹점 -> Commerce API (POST /charges) -> Charge 객체 생성
  2. 소비자 -> hosted_url 접속 -> 토큰/네트워크 선택
  3. 소비자 지갑 -> 온체인 트랜잭션 실행
  4. (선택) DEX를 통한 USDC 자동 환전 (Uniswap V3)
  5. 1% Commerce 수수료 자동 차감
  6. 가맹점 지갑에 정산
- **수수료/시간 정보**:
  - Coinbase 유저 -> Commerce 가맹점: 무료, 즉시 정산 (2026.01 업데이트)
  - 외부 지갑 -> Commerce 가맹점: 1% + 네트워크 가스비 (Base에서 ~$0.01)
  - DEX 자동 스왑 포함 시: 약 1.3% + 가스비 (Uniswap V3 0.3% 추가)
  - 정산 속도: Base 기준 약 200ms (최적), 실사용 약 2초
- **특이사항/제약사항**:
  - 지원 암호화폐: BTC, ETH, LTC, BCH, DOGE 등 100+ 종
  - 핵심 네트워크: Base (약 200ms, 가스비 약 $0.01)
  - RESTful API: Rate Limit 10,000 req/hr, 100 req/s burst
  - Webhooks: HMAC-SHA256 서명 검증

### 3.1.2 Commerce Payments Protocol (Authorize-Capture)

- **시나리오 이름**: 온체인 에스크로 기반 인가-캡처 결제
- **참여자**: 소비자, 오퍼레이터(Coinbase/Shopify), 가맹점, 에스크로 스마트 컨트랙트
- **단계별 흐름**:
  1. Authorization: 소비자 -> 결제 서명 -> 오퍼레이터 -> 에스크로 스마트 컨트랙트에 자금 예치
  2. Capture: 가맹점 -> 상품 준비 완료 후 캡처 요청 -> 에스크로에서 가맹점 지갑으로 이동
  3. Void: 캡처 전 취소 시 소비자에게 전액 반환
  4. Reclaim: 인가 만료 후 미캡처 시 소비자가 직접 자금 회수
- **수수료/시간 정보**:
  - 거래 수수료: 1%
  - 네트워크 가스비: ~$0.01 (Base)
- **특이사항/제약사항**:
  - Shopify와 공동 개발, 오픈소스
  - 부분 캡처(partial capture) 지원
  - 물리적 상품 배송이 필요한 이커머스에 적합
  - 실제 채택률 극히 낮음 ($1.2M/5,700개 가맹점, Shopify 전체의 0.1% 미만)

### 3.1.3 Shopify 네이티브 통합 결제

- **시나리오 이름**: Shopify USDC 결제
- **참여자**: Shopify 가맹점, 소비자, Base 네트워크
- **단계별 흐름**:
  1. Shopify 가맹점이 플러그인 활성화 (별도 게이트웨이 불필요, 15분 내 활성화)
  2. 소비자가 USDC로 결제 (Base 네트워크)
  3. Base 기반 즉시 정산
- **수수료/시간 정보**:
  - Shopify USDC 결제: 1% (Base 네트워크)
  - 외환 수수료 없음
- **특이사항/제약사항**:
  - 34개국 550만 가맹점 접근 가능
  - 현지 법정화폐 기본 수령 또는 USDC 직접 수령 선택 가능

---

## 3.2 정산(Settlement) 시나리오

### 3.2.1 Self-Managed (비수탁형) 정산

- **시나리오 이름**: 가맹점 자체 지갑 직접 정산
- **참여자**: 가맹점, 가맹점 자체 지갑 (12단어 시드 구문)
- **단계별 흐름**:
  1. 결제 완료 시 가맹점 자체 지갑에 직접 정산
  2. DEX를 통한 USDC 자동 스왑 (선택)
- **수수료/시간 정보**:
  - 1% 차감
  - Coinbase 유저 결제 시: 즉시
  - Base 네트워크 결제: 약 200ms~2초
  - 법정화폐 인출: 불가
- **특이사항/제약사항**:
  - 키 관리는 가맹점 책임 (분실 시 영구 손실)
  - 카운터파티 리스크 없음

### 3.2.2 Coinbase-Managed (수탁형) 정산

- **시나리오 이름**: Coinbase Exchange 연동 정산
- **참여자**: 가맹점, Coinbase Exchange
- **단계별 흐름**:
  1. 결제 완료 시 Coinbase Exchange 계정에 정산
  2. USD로 자동 환전 (선택)
  3. 연결 은행 계좌로 출금 (ACH/Wire)
- **수수료/시간 정보**:
  - 1% 차감
  - 법정화폐 인출: 1-3 영업일 (ACH/Wire)
  - 법정화폐 환전 스프레드: 추정 1-1.5%
- **특이사항/제약사항**:
  - 법정화폐(USD) 수령 가능
  - Coinbase가 키 관리
  - Coinbase 리스크 존재

### 3.2.3 Commerce Payments Protocol (하이브리드) 정산

- **시나리오 이름**: 에스크로 기반 캡처 정산
- **참여자**: 오퍼레이터(Coinbase/Shopify), 가맹점, 스마트 컨트랙트
- **단계별 흐름**:
  1. 자금은 스마트 컨트랙트에 보관
  2. 오퍼레이터가 결제 흐름을 운영 (자금 접근 불가)
  3. 가맹점의 캡처 요청 시점에 즉시 정산
- **수수료/시간 정보**:
  - 캡처 시 즉시 정산
- **특이사항/제약사항**:
  - 부분 캡처(partial capture) 지원

---

## 3.3 환불(Refund) 시나리오

### 3.3.1 Self-Managed 플랜 환불

- **시나리오 이름**: 비수탁형 수동 환불
- **참여자**: 가맹점, 소비자
- **단계별 흐름**:
  1. 소비자가 가맹점에 환불 요청 (오프체인 커뮤니케이션)
  2. 가맹점이 소비자의 수령 주소 확인
  3. 가맹점이 자체 지갑에서 직접 온체인 전송
  4. (결제와 동일 네트워크로 환불 권장)
- **수수료/시간 정보**:
  - 환불 수수료: 별도 청구 없음
  - 네트워크 가스비: 가맹점 부담
  - 기존 1% Commerce 수수료: 반환되지 않음
- **특이사항/제약사항**:
  - Commerce API 통한 자동 환불 불가
  - 대시보드에 환불 버튼 없음
  - 업계 최하위 수준의 환불 자동화

### 3.3.2 Coinbase-Managed 플랜 환불

- **시나리오 이름**: 수탁형 환불
- **참여자**: 가맹점, 소비자, Coinbase Exchange
- **단계별 흐름**:
  1. 소비자가 가맹점에 환불 요청
  2. 가맹점이 Coinbase Exchange 인터페이스에서 환불 처리
  3. (법정화폐로 환전된 경우) 재환전 필요
  4. Coinbase Exchange를 통해 소비자에게 자금 전송
- **수수료/시간 정보**:
  - 환전 손실: 결제 시점 대비 환율 차이 손실 가능
- **특이사항/제약사항**:
  - 법정화폐로 환전된 경우 재환전 필요

### 3.3.3 Commerce Payments Protocol -- Void

- **시나리오 이름**: 에스크로 캡처 전 전액 취소
- **참여자**: 오퍼레이터, 소비자, 에스크로 컨트랙트
- **단계별 흐름**:
  1. 오퍼레이터가 `void()` 호출
  2. 에스크로에서 소비자 지갑으로 전액 반환
- **수수료/시간 정보**:
  - 수수료 미발생
  - 가스비 외 비용 없음
- **특이사항/제약사항**:
  - 캡처 전에만 가능

### 3.3.4 Commerce Payments Protocol -- Refund

- **시나리오 이름**: 에스크로 캡처 후 환불
- **참여자**: 오퍼레이터, 소비자, OperatorRefundCollector
- **단계별 흐름**:
  1. 오퍼레이터가 `refund()` 호출 (refundExpiry 이전)
  2. OperatorRefundCollector에서 유동성 공급
  3. 부분/전액 환불 실행
- **수수료/시간 정보**:
  - 네트워크 가스비: 가맹점 부담
- **특이사항/제약사항**:
  - refundExpiry 이후에는 환불 불가 (최종 확정 상태)
  - 환불 유동성은 오퍼레이터가 별도로 공급해야 함
  - 아직 초기 채택 단계

### 3.3.5 Commerce Payments Protocol -- Reclaim

- **시나리오 이름**: 인가 만료 후 소비자 직접 회수
- **참여자**: 소비자, 에스크로 컨트랙트
- **단계별 흐름**:
  1. authExpiry 이후, 미캡처 상태에서
  2. 소비자가 직접 `reclaim()` 호출
  3. 오퍼레이터 개입 불필요 (신뢰 최소화 안전장치)
- **특이사항/제약사항**:
  - 소비자의 최후의 보호 수단

### 3.3.6 분쟁 해결

- **시나리오 이름**: 분쟁 해결 절차
- **참여자**: 소비자, 가맹점, Coinbase
- **단계별 흐름**:
  1. 소비자가 가맹점에 직접 연락하여 환불 협의
  2. 거래 ID, 고객 정보, 커뮤니케이션 기록 등 증빙 제출
  3. Coinbase에 공식 불만 접수 시, 45 영업일 이내 응답
- **특이사항/제약사항**:
  - 차지백(Chargeback) 없음
  - 블록체인 거래는 비가역적이므로 소비자 보호 장치 제한적
  - 환불 자동화는 경쟁사 대비 가장 뒤처진 영역

# 4. Stripe Crypto

Stripe의 크립토 결제 서비스 포트폴리오이다. Bridge+Tempo+Privy+Stripe의 수직 통합으로 가맹점의 크립토 노출을 완전히 제거하는 "zero-crypto 경험"이 핵심이다.

---

## 4.1 결제(Payment) 시나리오

### 4.1.1 Stablecoin Payments (가맹점 USDC 수취)

- **시나리오 이름**: 스테이블코인 결제 -> USD 자동 정산
- **참여자**: 고객(월렛 보유), Stripe(crypto.stripe.com), 가맹점, Bridge 엔진
- **단계별 흐름**:
  1. 가맹점 서버: `POST /v1/payment_intents` (payment_method_types: "crypto")
  2. Stripe가 `client_secret` 반환
  3. 프론트엔드에서 `confirmPayment()` 호출 -> crypto.stripe.com 리디렉션
  4. 고객이 월렛 연결 -> 토큰/체인 선택 -> 트랜잭션 서명
  5. Stripe가 온체인 제출 및 블록 확인 모니터링
  6. 확인 후 `payment_intent.succeeded` 웹훅 발송
  7. 가맹점 Stripe 잔고에 USD로 반영
- **수수료/시간 정보**:
  - 수수료: 1.5% (가맹점 부담, 가스비 포함)
  - 온체인 확인: Ethereum ~12초, Solana ~0.4초, Base ~2초, Polygon ~2초
  - 가맹점 정산: T+2 영업일 (USD 은행 출금)
- **특이사항/제약사항**:
  - 지원 스테이블코인: USDC (Ethereum, Solana, Polygon, Base), USDP (Ethereum, Solana), USDG (Ethereum)
  - 지원 월렛: 400개 이상 (MetaMask, Phantom 등)
  - 가맹점 요건: 현재 미국(US) 기반 비즈니스만 수취 가능
  - 고객 범위: 글로벌 (월렛 보유 고객이면 국적 무관)

### 4.1.2 구독 결제 (Stablecoin Subscription)

- **시나리오 이름**: 스마트 컨트랙트 기반 반복 결제
- **참여자**: 고객(월렛 보유), Stripe, 전용 스마트 컨트랙트
- **단계별 흐름**:
  1. 고객이 최초 1회 "사전 승인(pre-approval)" 서명
  2. 스마트 컨트랙트가 승인된 범위 내에서 반복 인출 권한 획득
  3. 이후 재서명 불필요, 자동 정기 결제 실행
- **수수료/시간 정보**:
  - 수수료: 1.5%
- **특이사항/제약사항**:
  - SaaS/구독 경제와 크립토 결제를 연결하는 업계 최초의 실질적 구현
  - 블록체인의 "매 트랜잭션 서명 필요" 한계를 해결

### 4.1.3 Pay with Crypto (Crypto.com 연동)

- **시나리오 이름**: 소비자 크립토 잔고 직접 결제
- **참여자**: 소비자(Crypto.com 사용자), Stripe 가맹점, Crypto.com
- **단계별 흐름**:
  1. 소비자가 Crypto.com 앱 내 크립토 잔고로 Stripe 가맹점에서 직접 결제
  2. 법정화폐로 자동 전환
  3. 가맹점에 정산
- **수수료/시간 정보**:
  - 수수료: 미공개 (표준 Stripe 가맹점 수수료 적용 추정)
- **특이사항/제약사항**:
  - 사전 환전 불필요
  - 차지백 없음 (가맹점 유리)
  - 미국 우선 론칭, 글로벌 확대 예정

### 4.1.4 x402 머신/AI 에이전트 결제

- **시나리오 이름**: HTTP 402 기반 머신 결제 (Stripe Facilitator)
- **참여자**: 클라이언트(AI 에이전트), 서버(API 제공자), Stripe Facilitator, Base L2
- **단계별 흐름**:
  1. 클라이언트가 유료 리소스 요청
  2. 서버가 HTTP 402 + 결제 상세 반환
  3. 클라이언트가 결제 서명 생성
  4. 인증 포함 재요청
  5. Stripe deposit address -> 블록체인 모니터링 -> Stripe 잔고 반영
- **수수료/시간 정보**:
  - 프로토콜 수수료: 0% 주장 (Stripe 정산 수수료 별도)
  - 최근 30일: 7,500만+ 트랜잭션, 2,400만+ 달러 볼륨
- **특이사항/제약사항**:
  - 3대 HTTP 헤더: PAYMENT-REQUIRED, PAYMENT-SIGNATURE, PAYMENT-RESPONSE
  - 지원 통화: USDC on Base
  - x402 Foundation: Coinbase, Cloudflare, Google, Visa 공동 관리

### 4.1.5 MPP (Machine Payments Protocol)

- **시나리오 이름**: Tempo 블록체인 기반 에이전트 자율 결제
- **참여자**: AI 에이전트, 서비스 제공자, Tempo L1, Stripe
- **단계별 흐름**:
  1. AI 에이전트가 PaymentIntents API 기반 결제 요청
  2. Shared Payment Tokens(SPTs)를 통한 에이전트 간 결제 권한 위임
  3. 배치 정산 처리
- **수수료/시간 정보**:
  - 요청당 거의 제로 수수료
  - 서브-100ms 지연시간
  - 배치 정산
- **특이사항/제약사항**:
  - 결제 수단: 스테이블코인(Tempo) + 카드(Stripe/Visa) + BTC Lightning + 커스텀
  - 2026.03 론칭 (Tempo 메인넷과 동시)
  - x402 대비: 대량 마이크로페이먼트, 장기 세션에 적합

### 4.1.6 Crypto Onramp (법정화폐 -> 크립토)

- **시나리오 이름**: 법정화폐로 크립토 구매
- **참여자**: 사용자, DApp, Stripe Onramp 위젯
- **단계별 흐름**:
  1. 임베디드 위젯 또는 Stripe-hosted 리디렉션
  2. 사용자가 결제 수단 선택 (카드, Apple Pay, ACH)
  3. 내장 본인인증(KYC) + Stripe Radar 사기 방지
  4. 법정화폐 -> 크립토 전환 완료
- **수수료/시간 정보**:
  - 수수료: 약 5% (카드), ~1.5% (ACH)
  - 예시: $100 USDC 구매 시 $4.99
- **특이사항/제약사항**:
  - Onramp만 지원, Offramp 미제공
  - 지원 크립토: USDC (5개 체인), ETH, MATIC, AVAX, XLM
  - 지원 국가: 미국 (하와이 제외) 및 EU

---

## 4.2 정산(Settlement) 시나리오

### 4.2.1 USDC -> USD 자동 정산 (Full Shielding)

- **시나리오 이름**: 가맹점 크립토 노출 완전 제거 정산
- **참여자**: 고객, Stripe, Bridge 오케스트레이션 엔진, 가맹점, 은행
- **단계별 흐름**:
  1. 고객이 스테이블코인 전송 -> 블록체인에서 트랜잭션 확인 (Ethereum ~12초, Solana ~0.4초, Base ~2초, Polygon ~2초)
  2. Bridge 엔진이 수취한 스테이블코인을 자동으로 USD로 전환 (1:1 페깅 기반, OTC/유동성 풀 활용)
  3. USD로 전환된 금액이 가맹점의 Stripe 잔고에 반영 (기존 카드 결제 잔고와 통합 관리)
  4. Stripe 표준 정산 주기에 따라 가맹점 은행 계좌로 ACH/SEPA 출금
- **수수료/시간 정보**:
  - 정산 수수료: 1.5% 트랜잭션 수수료에 포함 (별도 없음)
  - 정산 주기: T+2 영업일
- **특이사항/제약사항**:
  - 가맹점은 크립토를 전혀 보유/관리하지 않음
  - Stripe Legal 기준, 별도 Settlement Services Provider와 계약 체결 필요

### 4.2.2 USDC 직접 수취 (Stablecoin Financial Accounts)

- **시나리오 이름**: 스테이블코인 직접 보유 정산
- **참여자**: 가맹점, Stripe Stablecoin Financial Accounts
- **단계별 흐름**:
  1. 가맹점이 USD 전환 없이 스테이블코인을 직접 보유
  2. 다중 통화 잔고 관리
  3. 필요 시 법정화폐 레일(ACH, SEPA)로 전환
- **수수료/시간 정보**:
  - 전환 수수료: 법정화폐-스테이블코인 전환 시 발생
  - 지원 레일: ACH, SEPA (USD, EUR)
- **특이사항/제약사항**:
  - 지원 국가: 101개국
  - 지원 스테이블코인: USDC (Circle), USDB (Bridge)
  - 가상/실물 카드 발급 가능

---

## 4.3 환불(Refund) 시나리오

### 4.3.1 Stablecoin Payments 환불

- **시나리오 이름**: 스테이블코인 결제 온체인 환불
- **참여자**: 가맹점, Stripe, 고객(월렛)
- **단계별 흐름**:
  1. 가맹점이 환불 요청 -- API: `POST /v1/refunds` (charge=ch_xxx) 또는 Stripe 대시보드에서 직접 요청
  2. Stripe가 가맹점 잔고에서 차감 (USD)
  3. Stripe가 해당 금액을 원래 결제에 사용된 스테이블코인으로 구매 (USD -> 스테이블코인 전환)
  4. 고객의 원래 월렛 주소로 스테이블코인을 온체인 전송 (새 블록체인 트랜잭션 생성)
  5. 고객 월렛에 수신 확인
- **수수료/시간 정보**:
  - 환불 수수료: 별도 명시 없음 (Stripe 부담 추정)
- **특이사항/제약사항**:
  - 환불 통화: 원래 결제에 사용된 스테이블코인
  - 분쟁(Dispute): 미지원 -- Stripe의 Disputes 시스템이 스테이블코인 결제에는 미적용
  - 차지백: 없음 (블록체인 특성)
  - 트랜잭션 취소/역전: 불가 (온체인 확정 후 되돌릴 수 없음)
  - 잘못된 주소 송금: 복구 불가
  - 결제 시점의 크립토 가치를 수동으로 추적/관리해야 함

### 4.3.2 Pay with Crypto 환불

- **시나리오 이름**: Crypto.com 연동 결제 환불
- **참여자**: 가맹점, 고객
- **단계별 흐름**:
  1. 가맹점이 직접 환불 처리 필요
  2. 스테이블코인 결제 시: 고객의 원래 월렛으로 스테이블코인 반환
- **수수료/시간 정보**:
  - 미공개
- **특이사항/제약사항**:
  - 차지백 없음
  - 결제 시점의 크립토 가치를 수동으로 추적/관리

### 4.3.3 분쟁(Dispute) 처리

- **시나리오 이름**: 크립토 결제 분쟁 처리
- **참여자**: 가맹점, 고객
- **단계별 흐름**:
  - 모든 크립토 결제 서비스가 전통적 차지백/분쟁 시스템을 지원하지 않음
  - 가맹점이 자체적으로 환불 여부 결정
- **특이사항/제약사항**:
  - 기존 Stripe 카드 결제의 강력한 Disputes 시스템이 크립토에는 미적용
  - 차지백 부재는 사기성 분쟁에서 가맹점이 자유롭다는 의미
  - 소비자 관점에서 결제 후 구제 수단 없음
