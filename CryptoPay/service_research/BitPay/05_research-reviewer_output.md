# BitPay 딥리서치 최종 보고서

## Executive Summary

> - **BitPay는 암호화폐 결제 게이트웨이 시장 점유율 약 20%의 1위 사업자이나, PayPal(6.5억 사용자)과 Stripe(수백만 가맹점)의 시장 진입으로 중기적(3-5년) 포지션 방어가 최대 과제이다.**
> - **핵심 경쟁력은 38개국 법정화폐 직접 정산 + 환율 리스크 완전 흡수 + 엔터프라이즈급 컴플라이언스(FinCEN MSB 14년)로, 전통 비즈니스의 암호화폐 결제 수용 장벽을 낮추는 데 특화되어 있다.**
> - **추정 연매출 약 $60M(2024), 수익의 60%가 가맹점 결제 수수료에서 발생하며, FX 스프레드(약 16%)라는 은닉 수익원이 존재한다. 카드 프로그램 3년째 중단으로 소비자 사업은 사실상 위기 상태이다.**
> - **사용자 평점이 극단적으로 이원화(B2B 플랫폼 4.1~4.5점 vs Trustpilot 1.2점)되어 있으며, 이는 B2B 결제 인프라 기업이 B2C 서비스까지 확장하면서 발생한 구조적 자원 분산 문제를 반영한다.**
> - **수수료 경쟁력(업계 최고 수준 2%), 구독 결제 미지원, Lightning Network 미지원이 3대 전략적 취약점이며, 스테이블코인 B2B 결제(733% 성장)와 AI 에이전트 결제가 주요 신규 기회 영역이다.**

## 1. 서비스 개요

### 1.1 BitPay란?

BitPay는 2011년 5월 Tony Gallippi와 Stephen Pair가 설립한 세계 최대 암호화폐 결제 프로세서이다. 미국 조지아주 애틀랜타에 본사를 두고 있으며, 미국 재무부 FinCEN에 MSB(Money Services Business)로 등록되어 있다. 핵심 가치 제안은 가맹점이 암호화폐를 수신하면서도 법정화폐로 정산받을 수 있는 "변동성 제로" 결제 인프라이다.

## 2. 결제(Payment) 시나리오 상세

### 시나리오 A: Hosted Checkout

BitPay가 호스팅하는 결제 페이지로 고객을 리다이렉트하는 가장 간편한 통합 방식이다.

**Step-by-Step 흐름**:

1. 가맹점 서버가 `POST /invoices` API를 호출하여 인보이스를 생성한다
2. BitPay API가 Invoice를 생성하고(status: "new") Invoice URL을 반환한다
3. 가맹점이 고객을 Invoice URL로 리다이렉트한다
4. 고객이 BitPay Hosted Checkout 페이지에서 결제 통화를 선택하고 QR 코드를 스캔하거나 지갑 주소를 복사한다
5. 고객 지갑에서 블록체인 트랜잭션을 브로드캐스트한다
6. BitPay가 결제를 감지하고 Invoice status를 "paid"로 변경한다
7. 블록체인에서 1+ confirmations 후 "confirmed" 상태로 전환한다
8. 2+ confirmations 후 "complete" 상태로 전환한다
9. BitPay가 IPN(Webhook)을 가맹점의 notificationURL로 전송한다
10. 가맹점이 `GET /invoices/{id}`로 상태를 검증한 후 주문을 처리한다

> 주의: BitPay IPN은 서명되지 않으므로(unsigned), 가맹점은 반드시 API로 상태를 재검증해야 한다.

### 시나리오 B: API 직접 통합

서버 사이드에서 BitPay REST API를 직접 호출하여 인보이스를 생성하고, 결제 UI를 자체 구현하는 방식이다.

**Step-by-Step 흐름**:

1. 가맹점 서버가 merchant facade 토큰으로 API 인증을 설정한다
2. `POST /invoices`를 호출하여 인보이스를 생성한다 (price, currency, notificationURL, buyer 정보 포함)
3. BitPay API가 Invoice 객체를 반환한다 (id, url, addresses, paymentTotals 포함)
4. 가맹점 프론트엔드에서 자체 결제 UI에 QR/주소를 표시하거나 Invoice URL을 임베드한다
5. 고객이 선택한 암호화폐로 결제를 전송한다
6. BitPay가 결제 상태 변경 시 IPN을 notificationURL로 POST한다
7. 가맹점 서버가 IPN 수신 후 `GET /invoices/{id}`로 상태를 직접 검증한다
8. status가 "confirmed" 또는 "complete"이면 주문을 처리한다

**SDK 지원**: Node.js, PHP, Ruby, Python, Java, C#, Go 공식 SDK 제공.

### 시나리오 C: POS 결제

오프라인 매장에서 암호화폐 결제를 수신하는 시나리오이다.

**Step-by-Step 흐름**:

1. 직원이 BitPay Checkout 앱(iOS/Android) 또는 웹 기반 Quick Checkout에 접속한다
2. 결제 금액을 법정화폐 기준으로 입력한다 (예: $50.00)
3. 앱이 API를 호출하여 Invoice를 생성하고 QR 코드를 화면에 표시한다
4. 고객이 자신의 모바일 지갑 앱으로 QR 코드를 스캔한다
5. 고객 지갑에서 결제를 승인하고 블록체인 트랜잭션을 전송한다
6. BitPay 앱에서 실시간 결제 확인을 표시한다 (Paid -> Confirmed -> Complete)
7. 가맹점 대시보드에 거래 기록이 반영된다

Quick Checkout(웹 기반)은 모든 플랜에 포함되며 추가 비용이 없다.

### 시나리오 D: Payment Links / Buttons

코딩 없이 결제를 수신하는 방식이다.

**Payment Buttons**: BitPay 대시보드에서 상품명/가격/통화를 설정하여 HTML 코드 스니펫을 생성한 후, 웹사이트에 붙여넣기만 하면 된다. 클릭 시 Hosted Checkout으로 이동한다.

**Payment Links**: 대시보드에서 결제 링크(고유 URL)를 생성하여 이메일, SNS, 메신저 등으로 공유한다. 클릭 시 Hosted Checkout으로 이동한다.

| 플랫폼 | 통합 방식 |

### 수수료 구조

| 월간 처리량 | 거래 수수료 |

### Invoice 상태 라이프사이클

| 상태 | 설명 |
|------|------|
| **New** | 인보이스 생성 직후. 고객 미결제 |
| **Paid** | 결제 트랜잭션 감지됨 (미확인, 보증 아님) |
| **Confirmed** | 기본 블록 확인 완료 (BTC: 1 confirmation) |
| **Complete** | 충분한 블록 확인 완료 (BTC: 2 confirmations). 정산 크레딧 반영 |
| **Expired** | 결제 창(기본 15분) 내 미결제 시 |
| **Invalid** | 결제되었으나 1시간 내 블록체인 확인 실패 |
| **Underpaid** | 요청 금액 미달 결제 -> 자동 환불 처리 |
| **Overpaid** | 초과 결제 -> 초과분 자동 환불 |

총 8개 법정화폐, 38개국 직접 은행 입금 지원. 단, 1개 법정화폐만 설정 가능하다.

**Step-by-Step 흐름**:

1. 고객이 암호화폐로 BitPay Invoice를 결제한다
2. BitPay가 결제 접수 시점의 환율로 법정화폐 가치를 고정한다 (Exchange Rate Lock)
3. Invoice status가 "complete"가 되면 가맹점 계정 잔액에 크레딧이 반영된다
4. 매 영업일, 전일 처리 완료 건의 합산 정산이 실행된다 (최소 정산 금액 이상인 경우)
5. BitPay가 은행 이체(ACH/SEPA/Wire)를 실행한다
6. 1-2 영업일 내 가맹점 은행 계좌에 입금이 확인된다

### 시나리오 B: 암호화폐 정산

**정산 가능 암호화폐 (15종)**: BTC, BCH, ETH, LTC, DOGE, XRP, SHIB, USDC, USDT, EUROC, APE 등

**Step-by-Step 흐름**:

1. 고객이 임의의 지원 암호화폐로 결제한다
2. Invoice "complete" 후 가맹점 계정에 크레딧이 반영된다
3. 매일(Daily) 정산이 실행되어 지정된 지갑 주소로 전송된다
4. 약 1시간 이내에 가맹점 지갑에서 수신을 확인한다

233개국 지원. 은행 계좌가 불필요하므로 법정화폐 정산이 불가한 국가에서도 이용 가능하다.

### 시나리오 C: 분할 정산

법정화폐와 암호화폐를 혼합하여 정산 비율을 설정할 수 있다.

**예시**: 30% USD (은행 입금) + 50% BTC + 20% BCH

**제약 조건**:
- 법정화폐는 1개만 선택 가능
- 암호화폐는 최대 5개까지 선택 가능
- 비율 합계는 반드시 100%
- 가맹점 대시보드에서 비율 조정 가능

### 정산 주기/속도

| 정산 방식 | 주기 | 입금 소요 |
|----------|------|----------|
| 법정화폐 (은행) | 매 영업일 (Daily) | 1-2 영업일 (T+1~T+2) |
| 암호화폐 (지갑) | 매일 (Daily) | 약 1시간 |

### 수수료 구조 (총비용 시뮬레이션)

정산 수수료는 거래 수수료(1-2% + $0.25)에 포함되며, 별도 정산 수수료는 없다.

**월 $50,000 처리 가맹점의 총비용 시뮬레이션**:

```
거래 수수료: $50,000 x 2% = $1,000
건당 수수료: 62.5건 x $0.25 = $15.63 (평균 거래 $800 가정)
FX 스프레드 (은닉, 추정): $50,000 x 0.7% = $350
총 추정 비용: ~$1,366 (실효 수수료율 약 2.7%)
```

**월 $1,500,000 처리 가맹점의 총비용 시뮬레이션**:

```
거래 수수료: $1,500,000 x 1% = $15,000
건당 수수료: 1,875건 x $0.25 = $468.75
FX 스프레드 (은닉, 추정): $1,500,000 x 0.7% = $10,500
총 추정 비용: ~$25,969 (실효 수수료율 약 1.7%)
```

[주: FX 스프레드는 공식 수수료에 포함되지 않는 은닉 수익원이다. BitPay는 "승인 거래소의 Bid 가격" 기반으로 환율을 산정하며, 이는 시장 중간값보다 구조적으로 BitPay에 유리하다 (04_business-modeler). 신뢰도: 추정]

### 환율 결정 메커니즘

| 단계 | 방법 |
|------|------|
| 암호화폐 -> USD/EUR | 복수 승인 거래소의 매수 주문(Bids) 기반 가중 평균 |
| USD -> 기타 법정화폐 | openexchangerates.org (중간 시장 환율) |
| 변동성 조정 | BitPay 자체 알고리즘 ("시장 변동성과 유동성 깊이를 팩터링") |
| 환율 유효 시간 | 최대 15분 (인보이스 결제 창 내) |

## 4. 환불(Refund) 시나리오 상세

### 시나리오 A: 전액 환불

**Step-by-Step 흐름**:

1. 가맹점이 BitPay 대시보드에 로그인하여 "Payments" 탭으로 이동한다
2. 환불 대상 결제 건을 선택하고 "Refund"를 클릭한다
3. "Full Refund"를 선택한다
4. BitPay가 환불 요청을 접수하고 1-2 영업일 내 처리한다
5. 고객에게 환불 안내 이메일이 발송된다 (환불 수령용 링크 포함)
6. 고객이 이메일 내 링크를 클릭하여 환불 수령 페이지에 접속한다
7. 고객이 환불 받을 지갑 주소를 입력한다 (지정 기한 내)
8. BitPay가 고객 지정 지갑으로 환불 트랜잭션을 전송한다
9. 가맹점/고객이 환불 완료 알림을 수신한다 (트랜잭션 ID 포함)

**API를 통한 환불**: `POST /refunds` 호출 시 invoiceId, amount, currency를 지정한다. Webhook으로 환불 상태 변경 알림을 수신할 수 있다.

[주: 공식 소요 시간은 1-2 영업일이나, 실제로는 72시간~수주 지연 사례가 다수 보고되고 있다 (03_consumer-analyst)]

### 시나리오 B: 부분 환불

전액 환불과 동일한 흐름이나, 3단계에서 "Partial Refund"를 선택하고 금액을 직접 입력한다. 금액은 원본 인보이스의 pricing currency(예: USD) 기준으로 입력하며, 환불 시점 환율로 암호화폐 수량이 재계산된다.

### 환불 통화 처리

- **환불 통화**: 고객이 결제 시 사용한 동일 암호화폐로 환불
- **환불 금액 산정**: 원본 인보이스의 법정화폐(pricing currency) 기준 금액을 환불 시점 환율로 암호화폐 수량 재계산
- **예시**: $100 상당 BTC 결제 -> $50 부분 환불 요청 -> 환불 시점 BTC/USD 환율 기준 $50 상당 BTC 전송
- **네트워크 수수료**: 환불금에서 블록체인 마이너 비용이 공제된다. 소액 환불 시 공제 비율이 높아질 수 있다

### BitPay Guarantee

부정 차지백(fraudulent chargeback) 발생 시 적격 가맹점에 보상하는 프로그램이다.

- 가맹점이 주문 세부사항, 커뮤니케이션 기록, 배송 증빙 등을 제출
- BitPay가 결제 네트워크와 협력하여 차지백 취소 및 자금 반환 시도
- 주로 BitPay Card 관련 시나리오에서 적용. 순수 온체인 결제에는 차지백 자체가 구조적으로 존재하지 않음

### 분쟁 해결 절차

1. 고객이 가맹점에 직접 환불/분쟁을 요청한다 (BitPay는 중개자로서 직접 분쟁 해결 불가)
2. 가맹점이 내부 검토 후 환불 여부를 결정한다
3. 환불 승인 시 BitPay 대시보드/API로 환불을 처리한다. 거부 시 고객에게 사유를 통보한다
4. BitPay는 가맹점의 요청(authorization)이 있는 경우에만 환불을 촉진한다

> BitPay는 가맹점-고객 간 분쟁에서 중재자 역할을 하지 않는다. 기부금에 대해서는 환불 촉진 자체를 제공하지 않는다.

| 항목 | BitPay | Stripe Crypto | PayPal Crypto | CoinGate | BTCPay Server |
|------|--------|---------------|---------------|----------|---------------|
| 자동 환불 | O (과다/과소 결제) | **O (Stripe 표준)** | **O** | O | O (Pull Payments) |
| 부분 환불 | **O** | O | O | **O** | O |

### 5.4 포지셔닝 분석

BitPay는 **"엔터프라이즈 암호화폐 결제의 사실상 표준(de facto standard)"** 포지션을 점유하고 있다.

```
                        법정화폐 정산
                            |
                   BitPay   |   CoinGate
                     *      |     *
                            |
   높은 수수료 ─────────────┼──────────────── 낮은 수수료
                            |
                            |        BTCPay Server
                  Coinbase  |              *
                  Commerce  |
                     *      |
                            |
                       암호화폐 정산
```

- **BitPay**: 법정화폐 정산 + 높은 수수료. 전통 비즈니스 엔터프라이즈 타겟
- **CoinGate**: 법정화폐 정산 + 중간 수수료. EU MiCA 라이선스로 유럽 특화
- **BTCPay Server**: 암호화폐 정산 + 제로 수수료. 기술 역량 있는 프라이버시 중시 가맹점
- **Coinbase Commerce**: 암호화폐 정산 + 중간 수수료. Web3 네이티브 비즈니스

**가장 위협적인 경쟁사**:

1. **Stripe Crypto**: 기존 수백만 가맹점 기반 + 구독 결제 네이티브 + Tempo 자체 블록체인. 결제 정산 인프라 자체를 재정의하려는 전략
2. **PayPal Crypto**: 6.5억 사용자 + PYUSD 자체 스테이블코인 + 0.99% 공격적 수수료. 미국 시장에서 직접적 위협
3. **BTCPay Server**: 무료 오픈소스 + Lightning Network 지원 + 완전 비수탁. 기술 역량 있는 가맹점 이탈의 주요 대안

### 6.1 수익 구조

BitPay는 전통 결제 프로세서(Stripe, Square)와 유사한 양면 플랫폼 모델을 운영한다.

| FX 스프레드 (은닉 수익원) | ~$9.7M | 16% | 추정 (스프레드 크기 미공개) |
| BitPay Send 수수료 (1%) | ~$7.6M | 13% | 확인됨 (공식 가격) + 역산 |

| **합계** | **~$60M** | **100%** | 외부 분석 기관 추정치와 일치 |

> [주: FX 스프레드는 BitPay가 "승인 거래소 Bids"를 기준으로 환율을 산정하기 때문에 구조적으로 발생하는 은닉 수익원이다. 또한 "시장 변동성과 유동성 깊이를 팩터링"한다는 표현은 변동성이 큰 시기에 스프레드를 확대할 수 있는 재량권을 의미한다 (04_business-modeler)]

## 8. 전략적 시사점

### 핵심 발견 5가지

1. **"엔터프라이즈 결제 인프라" vs "소비자 플랫폼" 정체성 딜레마**: BitPay는 가맹점 서비스에서 업계 표준을 유지하고 있으나, 소비자 서비스(Wallet, Card)에서는 구조적으로 취약하다. 자원의 전략적 재배분 또는 소비자 사업 축소가 필요한 시점이다.

2. **수수료 방어력이 최대 취약점**: 저볼륨 가맹점 기준 2% + $0.25는 업계 최고 수준이다. BTCPay(0%), NOWPayments(0.5%), PayPal(0.99% 프로모션)의 하방 압력이 지속되고 있다. 여기에 FX 스프레드(추정 0.5-1.0%)까지 포함하면 실효 수수료율은 더 높아진다.

3. **PayPal/Stripe 진입이 중기적 생존 위협**: 두 사업자 모두 기존 수백만~수억 고객 기반에 암호화폐를 "추가 옵션"으로 제공하는 전략이며, BitPay의 핵심 고객("암호화폐 결제를 수신하고 싶은 전통 가맹점")이 정확히 이들의 타겟이다.

4. **카드 프로그램 3년 중단은 은행 관계 역량에 대한 의문**: Metropolitan Commercial Bank 철수 이후 새 은행 파트너를 확보하지 못한 것은 단순한 서비스 중단을 넘어 BitPay의 금융 인프라 구축 역량에 대한 구조적 의문을 제기한다.

5. **스테이블코인 결제 급성장이 최대 기회**: BitPay 내 스테이블코인 비중이 30%(2024) -> 40%(2025)로 증가하고 있으며, B2B 스테이블코인 결제가 733% 성장하고 있다. 이 트렌드와 기존 인프라의 결합이 성장의 핵심 동력이다.

### 기회 영역

| 기회 | 실현 가능성 | 근거 |
|------|-----------|------|
| **스테이블코인 B2B 결제 확대** | 높음 | 기존 인프라(BitPay Send) 활용 가능, 시장 733% 성장 |
| **구독/반복 결제 기능 추가** | 높음 | SaaS/AI 기업 니즈, 기존 인보이스 시스템 확장 가능 |
| **AI 에이전트 결제 API** | 중간 | 신시장이나 Coinbase/Cloudflare가 x402로 선점 중 |
| **APAC 법정화폐 정산 확대** | 중간 | 최고 성장률 지역이나 현지 은행 파트너십 필요 |
| **임베디드 파이낸스(APM 통합)** | 높음 | Nuvei, BlueSnap 파트너십 확대로 간접 가맹점 확보 |
| **크로스보더 B2B 정산** | 높음 | 38개국 정산 인프라 활용, SWIFT 대비 속도/비용 우위 |

### 위협 요인

| 위협 | 심각도 | 근거 |
|------|--------|------|
| **Stripe/PayPal 시장 진입** | CRITICAL | 기존 수백만/수억 가맹점 기반 레버리지, 낮은 전환 비용 |
| **수수료 하방 압력** | HIGH | BTCPay(0%), NOWPayments(0.5%), PayPal(0.99%) |
| **BTCPay/Lightning Network 성장** | HIGH | BTC 즉시 결제/초저비용, BitPay 중개 역할 축소 |
| **카드 프로그램 복구 실패** | HIGH | 소비자 사업 사실상 포기, 사용자 기반 이탈 가속 |
| **EU 시장 잠식 (CoinGate MiCA)** | MEDIUM | BitPay MiCA 미보유, CoinGate 유럽 선점 |

### 권고사항

1. **수수료 재구조화 (단기)**: 저볼륨 가맹점 대상 1.5% 이하로 인하하여 이탈 방지. Network Cost의 투명성 개선 (실제 온체인 비용과 서비스 수수료 분리 표기)

2. **구독 결제 기능 출시 (단기-중기)**: 스마트 컨트랙트 기반 반복 결제를 개발하여 Stripe Crypto에 대응. SaaS/AI 기업 시장 진출

3. **카드 프로그램 조속 재개 또는 명확한 포기 선언 (단기)**: 타임라인 없는 불확실성이 소비자 신뢰를 지속 훼손하고 있다. 재개 불가 시 소비자 사업 축소를 공식화하고 가맹점 인프라에 자원을 집중하는 것이 합리적

4. **고객 지원 체계 개편 (단기)**: 실시간 채팅/전화 지원 도입, 환불 프로세스 자동화. Trustpilot 1.2점 회복의 최소 조건

5. **MiCA 라이선스 취득 (중기)**: EU 시장에서 CoinGate 대응을 위한 규제 인프라 확보

6. **Lightning Network 지원 (중기)**: BTC 소액 결제 시장 진입, BTCPay Server 대비 경쟁력 확보
