## 3. 결제(Payment) 시나리오 상세

### 3.1 지원 암호화폐, 수수료, 처리 시간 (공통)

**지원 암호화폐 (결제 수신)**: 100+ 종
- 주요: BTC, BCH, ETH, LTC, DOGE, XRP, SHIB
- 스테이블코인: USDC, USDT, GUSD, PAX, BUSD, DAI, EUROC
- ERC-20 토큰 다수

**수수료 구조**:

| 월간 처리량 | 거래 수수료 |
|------------|-----------|
| $500,000 미만 | 2% + $0.25/건 |
| $500,000 -- $999,999 | 1.5% + $0.25/건 |
| $1,000,000 이상 | 1% + $0.25/건 |

- 별도 Network Cost(블록체인 마이너 비용): 인보이스 금액의 0.05% 미만 또는 $0.01 미만이면 미부과
- 월 구독료 없음 (사용량 기반 과금)

**처리 시간 (블록체인 확인)**:

| 암호화폐 | Confirmed 상태 | Complete 상태 |
|----------|---------------|--------------|
| BTC | 약 10분 (1 confirmation) | 약 20분 (2 confirmations) |
| ETH | 약 15초 -- 수분 | 네트워크 상태에 따라 상이 |
| 스테이블코인 (ERC-20) | ETH와 동일 | ETH와 동일 |
| LTC | 약 2.5분 | 약 5분 |
| XRP | 수초 | 수초 |

### 3.2 Hosted Checkout (결제 페이지)

BitPay가 호스팅하는 결제 페이지로 고객을 리다이렉트하는 방식. 가장 간편한 통합.

**Step-by-Step 흐름**:

```
[가맹점 서버] --> (1) POST /invoices API 호출 --> [BitPay API]
                                                      |
[BitPay API] --> (2) Invoice 생성 (status: "new") --> Invoice URL 반환
                                                      |
[가맹점] --> (3) 고객을 Invoice URL로 리다이렉트 --> [BitPay Hosted Checkout Page]
                                                      |
[고객] --> (4) 결제 통화 선택 --> QR 코드 스캔 또는 지갑 주소 복사
                                                      |
[고객 지갑] --> (5) 블록체인 트랜잭션 브로드캐스트 --> [블록체인 네트워크]
                                                      |
[BitPay] --> (6) 결제 감지, Invoice status "paid"로 변경
                                                      |
[블록체인] --> (7) 1+ confirmations --> Invoice status "confirmed"
                                                      |
[블록체인] --> (8) 2+ confirmations --> Invoice status "complete"
                                                      |
[BitPay] --> (9) IPN(Webhook) 전송 --> [가맹점 notificationURL]
                                                      |
[가맹점] --> (10) GET /invoices/{id}로 상태 검증 --> 주문 처리
```

**Invoice 상태 라이프사이클**:
1. **New**: 인보이스 생성 직후. 고객이 아직 결제하지 않은 상태.
2. **Paid**: 결제 트랜잭션 감지됨. 단, 결제 보증이 아님 (아직 미확인 트랜잭션).
3. **Confirmed**: 블록체인에서 기본 확인 완료 (BTC 기준 1 confirmation).
4. **Complete**: 충분한 블록 확인 완료 (BTC 기준 2 confirmations). 가맹점 계정에 정산 크레딧 반영.
5. **Expired**: 결제 창(기본 15분) 내 미결제 시.
6. **Invalid**: 결제되었으나 1시간 내 블록체인 확인 실패.

**예외 상태**:
- **Underpaid**: 요청 금액 미달 결제. 자동으로 고객에게 환불 처리.
- **Overpaid**: 초과 결제. 초과분은 자동으로 고객에게 환불 처리.

### 3.3 API 직접 통합 (결제 인보이스 생성)

서버 사이드에서 BitPay REST API를 직접 호출하여 인보이스를 생성하고, 결제 UI를 자체 구현.

**Step-by-Step 흐름**:

```
[가맹점 서버] --> (1) API 토큰 생성 (merchant facade)
                      |
[가맹점 서버] --> (2) POST /invoices 호출
                      {
                        price: 100.00,
                        currency: "USD",
                        notificationURL: "https://merchant.com/ipn",
                        redirectURL: "https://merchant.com/order/complete",
                        buyer: { name, email, ... }
                      }
                      |
[BitPay API] --> (3) Invoice 객체 반환 (id, url, addresses, paymentTotals...)
                      |
[가맹점 프론트엔드] --> (4) 자체 결제 UI에 QR/주소 표시 또는 BitPay Invoice URL 임베드
                      |
[고객] --> (5) 선택한 암호화폐로 결제 전송
                      |
[BitPay] --> (6) 결제 상태 변경 시 IPN POST to notificationURL
                      {
                        id: "invoice_id",
                        status: "paid" | "confirmed" | "complete",
                        ...
                      }
                      |
[가맹점 서버] --> (7) IPN 수신 후 GET /invoices/{id}로 상태 직접 검증
                      (BitPay는 IPN에 서명하지 않으므로, 반드시 API로 재검증 필수)
                      |
[가맹점 서버] --> (8) status "confirmed" 또는 "complete" 확인 후 주문 처리
```

**SDK 지원**: Node.js, PHP, Ruby, Python, Java, C#, Go 공식 SDK 제공.

### 3.4 POS 시스템 결제

## 4. 정산(Settlement) 시나리오 상세

### 4.1 법정화폐 직접 정산 (Bank Settlement)

**지원 법정화폐**:

| 통화 | 최소 정산 잔액 | 정산 후 입금 소요 |
|------|--------------|-----------------|
| USD | 미공개 (추정 $20 내외) | 1 영업일 |
| EUR | 100 EUR | 1-2 영업일 |
| GBP | 1,000 GBP | 2 영업일 |
| CAD | 5,000 CAD | 2 영업일 |
| AUD | 2,500 AUD | 2 영업일 |
| MXN | 미공개 | 2 영업일 |
| NZD | 미공개 | 2 영업일 |

- 총 **8개 법정화폐**, **38개국** 직접 은행 입금 지원
- 1개 법정화폐만 설정 가능 (예: USD 또는 EUR 중 택 1)

**법정화폐 정산 흐름**:

```
[고객] --> (1) 암호화폐로 BitPay Invoice 결제
                      |
[BitPay] --> (2) 결제 접수 시점의 환율로 법정화폐 가치 고정 (Exchange Rate Lock)
                      |
[BitPay] --> (3) Invoice status "complete" --> 가맹점 계정 잔액에 크레딧
                      |
[BitPay] --> (4) 매 영업일, 전일 처리 완료 건의 합산 정산 실행
                      (잔액이 최소 정산 금액 이상인 경우)
                      |
[BitPay] --> (5) 은행 이체(ACH/SEPA/Wire) 실행
                      |
[가맹점 은행 계좌] --> (6) 1-2 영업일 내 입금 확인
```

### 4.2 암호화폐 정산 옵션

**지원 정산 암호화폐 (15종)**:
- BTC, BCH, ETH, LTC, DOGE, XRP, SHIB
- USDC, USDT, EUROC
- APE 등

**암호화폐 정산 흐름**:

```
[고객] --> (1) 임의의 지원 암호화폐로 결제
                      |
[BitPay] --> (2) Invoice "complete" 후 가맹점 계정에 크레딧
                      |
[BitPay] --> (3) 매일(Daily) 정산 실행 --> 지정된 지갑 주소로 전송
                      |
[가맹점 지갑] --> (4) 약 1시간 이내 수신 확인
```

- 최대 **5개 암호화폐**로 분할 정산 가능
- 233개국 지원 (은행 계좌 불필요)
- 법정화폐 정산이 불가한 국가에서도 암호화폐 정산으로 서비스 이용 가능

### 4.3 분할 정산 (Split Settlement)

법정화폐와 암호화폐를 혼합하여 정산 비율 설정 가능.

**예시**:
- 30% USD (은행 입금) + 50% BTC + 20% BCH
- 가맹점 대시보드에서 비율 조정 가능

**제약 조건**:
- 법정화폐는 1개만 선택 가능
- 암호화폐는 최대 5개까지 선택 가능
- 비율 합계는 반드시 100%

### 4.4 정산 수수료 및 FX 처리

| 항목 | 내용 |
|------|------|
| 정산 수수료 | 거래 수수료(1-2%)에 포함, 별도 정산 수수료 없음 |
| FX 처리 | 결제 시점 환율 고정 (BitPay가 환율 리스크 흡수) |
| 환율 소스 | BitPay 자체 환율 (복수 거래소 중간값 기반, 추정) |
| Network Cost | 블록체인 마이너 비용, 소액 인보이스의 경우 미부과 |

### 4.5 정산 주기 요약

| 정산 방식 | 주기 | 최소 잔액 필요 | 입금 소요 |
|----------|------|--------------|----------|
| 법정화폐 (은행) | 매 영업일 (Daily) | 통화별 상이 | 1-2 영업일 |
| 암호화폐 (지갑) | 매일 (Daily) | 없음 (추정) | 약 1시간 |

## 5. 환불(Refund) 시나리오 상세

### 5.1 BitPay 환불 프로세스 개요

BitPay에서의 환불은 전통 결제와 다른 고유한 프로세스를 따름. 암호화폐 거래는 블록체인에서 비가역적이므로, 환불은 별도의 새 트랜잭션으로 실행됨.

### 5.2 가맹점 대시보드를 통한 환불

**Step-by-Step**:

```
[가맹점] --> (1) BitPay 대시보드 로그인 --> "Payments" 탭 이동
                      |
[가맹점] --> (2) 환불 대상 결제 건 선택 --> 우상단 "Refund" 클릭
                      |
[가맹점] --> (3) 환불 유형 선택:
                  - Full Refund: 전액 환불
                  - Partial Refund: 부분 환불 (금액 직접 입력)
                      |
[BitPay] --> (4) 환불 요청 접수 --> 1-2 영업일 내 처리
                      |
[BitPay] --> (5) 고객에게 환불 안내 이메일 발송
                  (환불 수령용 링크 포함)
                      |
[고객] --> (6) 이메일 내 링크 클릭 --> 환불 수령 페이지 접속
                      |
[고객] --> (7) 환불 받을 지갑 주소 입력
                  (지정 기한 내 입력 필요)
                      |
[BitPay] --> (8) 고객 지정 지갑으로 환불 트랜잭션 전송
                      |
[가맹점/고객] --> (9) 환불 완료 알림 수신 (트랜잭션 ID 포함)
```

### 5.3 API를 통한 환불

**Step-by-Step**:

```
[가맹점 서버] --> (1) merchant facade 토큰으로 인증
                      |
[가맹점 서버] --> (2) POST /refunds 호출
                      {
                        invoiceId: "원본_인보이스_ID",
                        amount: 50.00,        // 원본 인보이스 통화 기준
                        currency: "USD",       // 원본 인보이스의 pricing currency
                        preview: false
                      }
                      |
[BitPay API] --> (3) 환불 요청 객체 반환 (refund ID, status, amount...)
                      |
[BitPay] --> (4) 내부 처리 (1-2 영업일)
                      |
[BitPay] --> (5) 고객 이메일로 환불 안내 발송
                      |
[고객] --> (6) 지갑 주소 제공 --> 환불 수령
                      |
[BitPay] --> (7) 환불 상태 변경 시 Webhook 전송 (선택적)
```

### 5.4 전액 환불 / 부분 환불

| 유형 | 설명 | 통화 기준 |
|------|------|----------|
| 전액 환불 (Full Refund) | 인보이스 전체 금액 환불 | 원본 인보이스의 pricing currency (예: USD) |
| 부분 환불 (Partial Refund) | 지정 금액만 환불 | 원본 인보이스의 pricing currency |
| 과다결제 환불 (Overpayment) | 초과분 자동 환불 | 결제 암호화폐 기준 |
| 과소결제 환불 (Underpayment) | 미달 금액 자동 환불 | 결제 암호화폐 기준 |

### 5.5 환불 통화 처리

- **환불 통화**: 고객이 결제 시 사용한 동일 암호화폐로 환불
- **환불 금액 산정**: 원본 인보이스의 법정화폐(pricing currency) 기준 금액을 환불 시점 환율로 암호화폐 수량 재계산
- **예시**: 고객이 $100 상당의 BTC로 결제 -> 부분 환불 $50 요청 -> 환불 시점 BTC/USD 환율 기준 $50 상당 BTC 전송
- **환불 준비금(Refund Reserve)**: 법정화폐(fiat) 기준으로만 설정/유지

### 5.6 분쟁 해결 절차

암호화폐 결제는 전통 카드 결제와 달리 차지백(chargeback)이 존재하지 않음. 블록체인 트랜잭션은 비가역적이므로 분쟁 해결 체계가 상이함.

**분쟁 해결 프로세스**:

```
[고객] --> (1) 가맹점에 직접 환불/분쟁 요청 (BitPay는 중개자, 직접 분쟁 해결 불가)
                      |
[가맹점] --> (2) 내부 검토 후 환불 여부 결정
                      |
[가맹점] --> (3-A) 환불 승인 시: BitPay 대시보드/API로 환불 처리
              (3-B) 환불 거부 시: 고객에게 사유 통보
                      |
[BitPay] --> (4) 가맹점 요청 시에만 환불 촉진 (BitPay 자체 판정 없음)
```

- BitPay는 가맹점의 요청(authorization)이 있는 경우에만 환불을 촉진(facilitate)
- BitPay는 가맹점과 고객 간 분쟁에서 중재자 역할을 하지 않음
- 기부금(donation)에 대해서는 환불 촉진 자체를 제공하지 않음

### 5.7 BitPay Guarantee 프로그램

BitPay Guarantee는 적격 가맹점에 대한 추가적인 차지백 보호 프로그램.

**주요 내용**:
- 부정 차지백(fraudulent chargeback) 발생 시 적격 가맹점에 보상
- 가맹점이 주문 세부사항, 커뮤니케이션 기록, 배송 증빙 등을 제출
- BitPay가 결제 네트워크와 협력하여 차지백 취소 및 자금 반환 시도
- 전통 결제 대비 차지백 자체가 희소 (암호화폐 거래의 비가역성)
- 분쟁 예방을 위한 가이드라인 및 교육 자료 제공

> 참고: 이 프로그램은 주로 BitPay Card나 기타 카드 결제 관련 시나리오에서 적용되며, 순수 온체인 암호화폐 결제에는 차지백 자체가 구조적으로 존재하지 않음.

### 6.2 BitPay의 경쟁 우위

1. **시장 선점자 이점 (First-Mover Advantage)**: 2011년 설립, 14년 이상의 운영 이력. 가장 오래된 암호화폐 결제 프로세서.
2. **엔터프라이즈급 인프라**: Microsoft, AMC Theatres 등 대형 기업 고객 확보. PCI 컴플라이언트, 회계 소프트웨어(QuickBooks/Xero) 연동.
3. **법정화폐 직접 정산**: 38개국 은행 직접 입금. 가맹점의 암호화폐 보유 리스크 완전 제거.
4. **통합 생태계**: 결제 처리 + 지갑 + 직불카드 + 대량 지급을 하나의 플랫폼에서 제공.
5. **규제 준수**: FinCEN 등록 MSB, PCI-DSS 준수, 강력한 KYC/AML 체계.

### 6.3 BitPay의 약점 및 위협

1. **수수료 경쟁력**: BTCPay(무료), NOWPayments(0.5%), Binance Pay(무료) 대비 높은 수수료(1-2%).
2. **지원 코인 수**: NOWPayments(350+) 대비 제한적(100+).
3. **탈중앙화 트렌드**: BTCPay Server 등 셀프호스팅/오픈소스 솔루션의 성장. 프라이버시 중시 사용자의 이탈 가능성.
4. **생태계 의존성**: Coinbase Commerce(Coinbase 생태계), Binance Pay(Binance 생태계)의 빌트인 사용자 기반 위협.
5. **Lightning Network**: BTC 소액결제의 즉시 확인/초저비용 솔루션 확산. Strike 등 Lightning 특화 서비스의 성장.

### 6.4 전략적 포지션 요약

BitPay는 **"엔터프라이즈 암호화폐 결제의 표준(de facto standard)"** 포지션을 점유. 대형 기업의 법정화폐 정산 니즈를 충족하는 풀스택 솔루션으로, 규제 준수와 안정성을 최우선시하는 가맹점 시장에서 강점을 유지. 다만, 수수료 경쟁력과 코인 다양성 측면에서 후발 주자들의 도전에 직면하고 있으며, 오픈소스/탈중앙화 솔루션의 성장도 중장기적 위협 요인.
